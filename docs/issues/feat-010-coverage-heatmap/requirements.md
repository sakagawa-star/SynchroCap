対象: feat-010
作成日: 2026-03-10

## 1. プロジェクト概要

### 1.1 何を作るか

feat-009で蓄積されたキャプチャデータ（image_points）を使い、画像領域全体に対するコーナー検出のカバレッジをヒートマップとして可視化する。ヒートマップはライブビュー上に半透明オーバーレイとして重畳表示する。

### 1.2 なぜ作るか

キャリブレーション精度を高めるには、画像全域にわたって均一にコーナーが検出されている必要がある。現状ではどの領域にコーナーが不足しているか視覚的に確認できない。ヒートマップにより、ユーザーはボードを配置すべき領域を直感的に把握できる。

### 1.3 誰が使うか

SynchroCapを使用してモーションキャプチャ用の同期録画を行うオペレーター。

### 1.4 どこで使うか

SynchroCapと同一のPC環境（Ubuntu Linux、micromamba SynchroCap環境）。カメラ（DFK33GR0234）が物理接続された状態で使用する。

---

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| カバレッジ | 画像領域のうち、キャプチャ済みコーナーが存在する範囲の広がり |
| ヒートマップ | 各コーナー点にガウシアンカーネルを加算し、カバレッジの密度を色で表現した可視化画像 |
| ガウシアンカーネル | 各コーナー点の影響範囲を広げるための正規分布フィルタ。σ=画像幅×5% |
| image_points | 画像上で検出されたコーナーの2D座標。shape=(N,1,2), float32 |
| オーバーレイ | ライブビューのフレーム上に半透明で重畳表示される画像 |

---

## 3. 機能要求一覧

### FR-001: ヒートマップ生成

- **概要**: 全キャプチャの image_points を集約し、各コーナー点にガウシアンカーネルを加算してカバレッジ密度をヒートマップ画像として生成する
- **入力**:
  - `captures: list[CaptureData]` — キャプチャリスト（1件以上）
  - `image_size: tuple[int, int]` — 画像サイズ (width, height)
- **処理**:
  1. 全キャプチャの `image_points`（shape=(N,1,2)）から2D座標を抽出し、1つの配列に結合する
  2. 画像サイズと同じ大きさの累積マップ（float32、初期値0）を用意する
  3. 各コーナー点の位置にガウシアンカーネルを加算する。ガウシアンの標準偏差σは画像幅の5%とする（例: 1920px → σ=96px）。これによりコーナー間の隙間が埋まり、ボードがカバーした領域が「面」として表示される
  4. 累積マップの最大値で正規化し、0〜255の範囲にスケーリングする（最大値が0の場合は全ピクセル0とする）
  5. `cv2.applyColorMap()` で `COLORMAP_TURBO` を適用し、BGR画像を生成する（TURBOはJETより知覚均一性が高く、段階の違いが視認しやすい）
  6. 累積マップの値が0のピクセルは黒（0,0,0）にする（カラーマップの青が「カバレッジなし」と混同されるのを防ぐ）
- **出力**: ヒートマップ画像（numpy.ndarray, shape=(height, width, 3), dtype=uint8, BGR）
- **受け入れ基準**:
  - キャプチャが1件以上ある状態でヒートマップが生成できる
  - ボードがカバーした領域が面として表示される（コーナー間の隙間が埋まる）
  - コーナーが多い領域ほど暖色（赤）、少ない領域ほど寒色（青）で表示される
  - カバレッジがないピクセルは黒で表示される（＝次にボードを置くべき場所）
  - 画像サイズと同じ解像度のヒートマップ画像が生成される

### FR-002: ライブビューへの自動オーバーレイ表示

- **概要**: キャプチャが1件以上存在する場合、ヒートマップをライブビュー上に半透明オーバーレイとして常時自動表示する。ユーザー操作（ボタン押下）は不要
- **入力**: なし（自動表示）
- **前提条件**: キャプチャが1件以上存在し、ヒートマップキャッシュが生成済みであること
- **処理**:
  1. 毎フレームの `_process_latest_frame()` でヒートマップキャッシュが存在する場合、ライブビューフレーム（またはボード検出オーバーレイ済みフレーム）にヒートマップを `cv2.addWeighted()` でアルファブレンドする。アルファ値は0.3（ヒートマップ側）、ベータ値は0.7（ライブビュー側）とする
  2. キャプチャが0件（キャッシュが `None`）の場合、オーバーレイは行わない（通常のライブビュー表示）
  3. ヒートマップ画像はキャプチャ追加・削除時にキャッシュを更新する。毎フレーム再計算しない
- **出力**:
  - キャプチャ1件以上: ライブビューにヒートマップが半透明で常時重畳表示される
  - キャプチャ0件: 通常のライブビュー表示
- **受け入れ基準**:
  - キャプチャが追加された瞬間に、ヒートマップが自動的にライブビューに表示される
  - ライブビューの映像が透けて見える（アルファ0.3）
  - ボード検出オーバーレイ（コーナーマーカー描画）とヒートマップオーバーレイが同時に表示される
  - ライブビューのFPS（30FPS）が維持される
  - ユーザーによるボタン操作は一切不要

### FR-003: ヒートマップキャッシュの更新タイミング

- **概要**: ヒートマップ画像の再計算タイミングを制御する
- **入力**: キャプチャの追加（自動キャプチャ成功）、削除（Delete/Clear All）
- **処理**:
  1. キャプチャ追加時: ヒートマップキャッシュを再計算する
  2. キャプチャ削除時（Delete, Clear All）: ヒートマップキャッシュを再計算する。キャプチャが0件になった場合はキャッシュを `None` にクリアする
  3. カメラ切替・タブ離脱時: キャプチャクリアに伴いキャッシュも `None` にクリアする
- **出力**: 更新されたヒートマップキャッシュ（numpy.ndarray | None）
- **受け入れ基準**:
  - 新しいキャプチャが追加されると、ヒートマップにその領域が反映される
  - キャプチャを削除すると、ヒートマップからその領域が除去される
  - Clear Allでヒートマップが消える

## 4. 非機能要求

### 4.1 パフォーマンス

| 項目 | 基準 |
|------|------|
| ヒートマップ生成 | 1回の生成が10ms以下（HD解像度、30キャプチャ、GaussianBlur + applyColorMap）|
| オーバーレイ合成 | `cv2.addWeighted()` は1回あたり1ms以下。毎フレーム実行しても30FPSを維持できる |
| キャッシュ戦略 | ヒートマップ画像はキャプチャ変化時にのみ再計算する。毎フレーム再計算しない |

### 4.2 信頼性

ヒートマップはメモリ上のキャッシュであり、キャプチャデータから随時再計算可能。永続化は不要。

### 4.3 ログ出力

Python標準の `logging` モジュールを使用する。ログ出力の詳細（レベル、出力ポイント）は機能設計書セクション8で定義する。

### 4.4 対応環境

Ubuntu Linux、micromamba SynchroCap環境。カメラ（DFK33GR0234）が物理接続された状態で使用する（セクション1.4と同一）。

## 5. 制約条件

### 5.1 使用必須ライブラリ

feat-009と同一。新規ライブラリの追加なし。`cv2.GaussianBlur()`、`cv2.applyColorMap()`、`cv2.addWeighted()` は既存ライブラリの機能。

### 5.2 カメラ設定変更禁止

feat-009と同一。Calibrationタブはカメラ設定を変更しない。

### 5.3 入力データ形式

feat-009で定義された `CaptureData` をそのまま使用する:

- `image_points`: numpy.ndarray, shape=(N,1,2), float32
- `image_size`: tuple[int, int] (width, height) — `_capture_image_size`

`CaptureData` のフィールドは変更しない。
