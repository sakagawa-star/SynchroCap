# investigation.md: feat-010 手動テストフィードバック

## イテレーション1 (2026-03-10)

### 問題1: ヒートマップの可読性が低い

#### 1.1 不具合の特定

- **対応する要求ID**: FR-001（ヒートマップ生成）、FR-002（オーバーレイ表示）
- **対応する設計セクション**: design.md 4.1、4.2
- **現在の動作**: 24コーナーが狭い領域に集中しているのに、4セルに分散して表示される。色の意味が不明で、1回のキャプチャの結果が良いのか悪いのかが判断できない。スクリーンショットでは、ボードが置かれた狭い領域に赤・緑・青・黒のブロックが混在している
- **期待する動作**: 「コーナーが多い領域ほど暖色（赤）、少ない領域ほど寒色（青）」（FR-001受け入れ基準）だが、1回のキャプチャで24コーナーが4セルに分散すると、最大値セル=赤、他=青/緑/黒になり、カバレッジの良し悪しが直感的に分からない

#### 1.2 原因分析

- **原因箇所**: `coverage_heatmap.py` の `generate()` メソッド
- **原因の説明**: CELL_SIZE_TARGET=80px でグリッド分割しているため、ボードが置かれた小さな領域が複数セルに跨ぎ、セルごとのコーナー数の差で色が分かれる。さらに最大値正規化のため、1回のキャプチャではコントラストが極端になる（最大セル=赤、1コーナーセル=ほぼ青）
- **根本原因**: グリッドセルサイズが大きすぎて解像度不足。加えて、JETカラーマップの値0=黒、値1=青の区別が視覚的に分かりにくい

#### 1.3 修正内容

**変更対象ファイル:**

1. `docs/issues/feat-010-coverage-heatmap/requirements.md` — FR-001の修正
   - CELL_SIZE_TARGET を 80px → 40px に変更（グリッド解像度を2倍に。HD 1920x1080 → 48x27グリッド）
   - カラーマップを `COLORMAP_JET` → `COLORMAP_TURBO` に変更（JETより知覚均一性が高く、段階の違いが分かりやすい）

2. `docs/issues/feat-010-coverage-heatmap/design.md` — セクション4.1の修正
   - CELL_SIZE_TARGET の変更を反映
   - カラーマップの変更を反映
   - 設計判断を追記

3. `src/synchroCap/coverage_heatmap.py` — 実装の修正
   - `CELL_SIZE_TARGET = 80` → `CELL_SIZE_TARGET = 40`
   - `cv2.COLORMAP_JET` → `cv2.COLORMAP_TURBO`

**変更しないファイル:**
- `ui_calibration.py` — ヒートマップ生成の呼び出し方は変更なし

#### 1.4 影響範囲

- テスト `test_coverage_heatmap.py` — グリッド数の期待値が変わるため、テストの更新が必要
- 他の機能への影響なし

#### 1.5 確認方法

- 手動テスト: 1回のキャプチャでヒートマップを表示し、ボード位置が色付き、それ以外が黒で、色の意味が直感的に分かること
- 自動テスト: グリッド計算テストの期待値を更新して実行

## イテレーション2 (2026-03-10)

### 問題: ヒートマップがコーナー点のみを表示し、ボードがカバーした領域が面として見えない

#### 1.1 不具合の特定

- **対応する要求ID**: FR-001（ヒートマップ生成）
- **対応する設計セクション**: design.md 4.1
- **現在の動作**: 各コーナー点をグリッドセルにカウントしているだけのため、コーナー間の隙間が黒く表示される。24コーナーがバラバラの赤い点として表示され、ボードがカバーした「領域」が面として認識できない
- **期待する動作**: ボードがカバーした領域が面として見え、複数キャプチャを重ねると画像全域がどの程度カバーされているかが直感的に分かる。画像の端・四隅に黒い領域が残っていれば「次はそこにボードを置くべき」と判断できる

#### 1.2 原因分析

- **原因箇所**: `coverage_heatmap.py` の `generate()` メソッド
- **原因の説明**: `numpy.histogram2d()` でコーナー点を直接グリッドセルにカウントしている。コーナー点は離散的（ボード上の交点のみ）であり、セルサイズを小さくするほどコーナー間の隙間が黒くなる。セルサイズを大きくすると解像度が下がり不正確になる。どちらにしてもコーナー点の離散性が問題
- **根本原因**: コーナー点の離散性を考慮せず、点の密度だけでヒートマップを生成していた。ヒートマップの目的（キャリブレーション用カバレッジ）を理解せず設計した

#### 1.3 修正内容

**方針**: 各コーナー点に対してガウシアンカーネル（σ=画像幅の5%）で影響範囲を広げてからヒートマップ化する。これにより、コーナー間の隙間が埋まり、ボードがカバーした領域が「面」として表示される。

**変更対象ファイル:**

1. `docs/issues/feat-010-coverage-heatmap/requirements.md` — FR-001の処理を改訂
2. `docs/issues/feat-010-coverage-heatmap/design.md` — セクション4.1の処理を改訂
3. `src/synchroCap/coverage_heatmap.py` — generate() メソッドをガウシアン累積方式に変更
4. `tests/test_coverage_heatmap.py` — グリッドベースのテストをガウシアン方式に合わせて更新

**アルゴリズム変更:**
- 変更前: `numpy.histogram2d()` でグリッドセルにカウント → カラーマップ適用
- 変更後: 画像サイズの累積マップ（float32）を用意 → 各コーナー点の位置にガウシアンカーネル（σ=画像幅の5%）を加算 → 最大値正規化 → カラーマップ適用 → 値0のピクセルは黒

**CELL_SIZE_TARGET は不要になる**（グリッド分割を廃止）。

**変更しないファイル:**
- `ui_calibration.py` — ヒートマップ生成の呼び出しインターフェース（`generate(points)` → BGR画像）は変更なし

#### 1.4 影響範囲

- `coverage_heatmap.py` のインターフェース変更なし（入力: points shape=(M,2)、出力: BGR画像）
- `ui_calibration.py` の呼び出し側は変更なし
- テストは内部ロジック変更に合わせて更新が必要

#### 1.5 確認方法

- 手動テスト: 1回のキャプチャでボードがカバーした領域が面として表示される
- 手動テスト: 複数キャプチャで画像全域のカバレッジが確認できる
- 手動テスト: 画像端の未カバー領域が黒で表示され、次にボードを置くべき場所が分かる
- 自動テスト: ガウシアン方式のテストに更新して実行
