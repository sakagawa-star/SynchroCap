# ChArUco ボード撮影ガイド（intrinsics キャリブレーション用）

**位置づけ**: 参考資料（feat-021）。SynchroCap のキャリブレーション機能（Tab5）または `tools/offline_calibration.py` 向けに ChArUco ボードを撮影する際のガイド。将来のガイダンス機能（UI 表示・自動診断）の元ネタとなる。

**作成日**: 2026-06-10
**実証データ**: cam41520554（DFK 33GR0234, 3.5mm レンズ, 1920x1080, ChArUco 5x7 / square 34mm / marker 22mm / DICT_6X6_250, A4 ボード）

---

## 1. 背景: RMS が良くても fx が決まらない問題

RMS 再投影誤差は「キャリブレーションが成功したか」の指標として**不十分**である。実例:

| | 失敗データ (20260610-132944) | 改善データ (20260610-155344) |
|---|---|---|
| 枚数 | 88枚 | 50枚 |
| RMS 再投影誤差 | 0.4614 px（一見良好） | 0.2024 px |
| ボード傾き 中央値 / 75%点 | 13.5° / 17.9°（ほぼ正対） | 22.2° / 36.0° |
| ボード距離 | 0.41〜0.77 m | 0.38〜0.80 m |
| 推定 fx | 1112.6（**真値から −8%**） | 1216.9 |
| fx を ±8% 変えて固定した時の RMS 悪化 | **+11%（区別不能）** | 約2倍（明確に区別可能） |
| split-half の fx ばらつき | 2.6%（1108.8 / 1138.0） | 0.4% 以内 |
| 下流の外部パラメーター推定 RMSE | **142.74 px（破綻）** | **4.84 px（優秀）** |

- 改善データの fx=1216.9 は、独立した手法（部屋の基準点 + K 自由推定）の fx≈1210.5 と 0.5% で一致しており、真値に近いことが裏付けられている
- 過去データ（2026-04-18）でも同日2回のキャリブレーションで fx が 7〜9% 振れていた（cam41520557: 1123↔1220、cam41520556: 1172↔1089）。同じ原因（正対撮影）とみられる
- 8% ずれた fx は、K 既知の solvePnP 外部推定で矛盾を生み、RANSAC（閾値8px）が偏った inlier 集合を選ぶことで RMSE 142px 級の破綻に増幅される

## 2. 原理: なぜ傾きが必要か（fx-距離縮退）

正対した平面ボードの写る大きさは `fx × ボード実寸 ÷ 距離` だけで決まる。**fx を 9% 大きくし距離も 9% 遠ざけると画像は同一**になるため、距離（tvec）が未知数であるキャリブレーションでは fx と距離を分離できない（縮退）。

ボードを傾けると、1枚のボードの中に「カメラに近い側」と「遠い側」が同時にでき、手前のマスは大きく・奥のマスは小さい**台形**に写る。

```
   正対（等間隔）            傾き45°（手前→奥でマスが縮む）

  ┌──┬──┬──┬──┐            ┌────┬───┬──┬─┐
  ├──┼──┼──┼──┤            ├────┼───┼──┼─┤
  └──┴──┴──┴──┘            └────┴───┴──┴─┘
                              手前         奥
```

- ボード実寸は既知なので、傾き角に応じた**手前端と奥端の奥行き差は実寸で固定**される
- カメラを遠ざけるとこの縮み方の比率は変わる（望遠の「圧縮効果」と同じ）
- よって「全体の大きさ」+「台形の縮み具合」の2情報から fx と距離を分離できる

傾きが浅いと情報が検出ノイズに埋もれる。fx の 8% 差が生む画像上の差は、傾き 13.5° では約 0.4px（コーナー検出ノイズ 0.3〜0.5px と同レベル → 区別不能）、傾き 45° では数 px（明確に区別可能）。

**注意**: ロール（画面内でボードをクルクル回す）は遠近を生まないため fx には**無効**。有効なのはヨー（左右に振る）とピッチ（上下に倒す）。

## 3. 撮影ガイド

### 原則（3つ）

1. **傾き 30〜45° が主役** — fx/fy を決める唯一の情報源
2. **画面の全域に配置** — 歪み係数 k1, k2 は画像の端・四隅のデータからしか決まらない
3. **距離は変えなくてよい** — 傾けたボード1枚が近い点と遠い点を同時に提供する。検出可能距離の範囲内で固定で構わない

### 推奨撮影シーケンス（約30〜50枚）

| ステップ | ボードの画面内位置 | 傾き | 枚数 | 主な目的 |
|---|---|---|---|---|
| 1 | 中央 | 正対 (0°) | 2 | 基準・検出確認 |
| 2 | 中央 | 左ヨー 30–45° / 右ヨー 30–45° | 各2 | **fx** |
| 3 | 中央 | 上ピッチ 30–45° / 下ピッチ 30–45° | 各2 | **fy**・主点 |
| 4 | 中央 | 対角傾き（左上倒し / 右下倒し 等） | 4 | 主点・接線歪みの分離 |
| 5 | 四隅（各隅） | 20–30° 任意方向 | 各2 = 8 | **歪み係数** |
| 6 | 四辺の中央（上下左右） | 30–45°（辺に沿う軸で） | 各1–2 = 4–8 | 歪み係数 + fx |
| 7 | （可能なら）近め / 遠め | 30° 程度 | 各2 | 補強（必須ではない） |

### 数値基準（受け入れ条件）

- **傾き 30° 以上のキャプチャが全体の3割以上、傾きの75%点が 35° 以上**（実証: 30°以上 15/50枚 + 75%点 36.0° で fx が決定できた。当初案の「過半数」までは不要）
- 傾き 60° 超はマーカー検出が落ちるため不要。狙うのは 30〜45°
- マーカーの画像上サイズ ≥ 32px（DICT_6X6 は境界込み 8×8 セルで、1セル約 4px が検出限界。A4 / marker 22mm / fx≈1200 では距離 ≤ 0.8m に相当）
- 傾けて一部のコーナーが検出できなくても問題ない（写った分だけ有効。最低6点・非共線は bug-009 で保護済み）
- 照明のテカリに注意（傾けると反射角が変わり、特定ポーズだけ検出が落ちる）

### ボードの物理条件

- **剛板に全面貼りすること**（アルミ複合板・MDF 等）。反り・たわみは fx の系統誤差に直結し、大判ほど反りやすい。紙のままは不可
- 印刷スケールで square / marker の実寸が公称からずれるため、**印刷後に定規で実測した値**を `--square-mm` / `--marker-mm` に渡す（intrinsics には square の絶対値はほぼ効かないが、外部キャリブレーションに流用する場合は効く）

### ボードサイズの指針（5x7 マス共通、fx≈1200・距離 0.54m で試算）

| | A4 (square 34mm) | A3 (≈48mm) | A1 (≈96mm) |
|---|---|---|---|
| 画面幅に占める割合 @0.54m | 約 20% | 約 28% | 約 56%（縦ははみ出す） |
| 45° 傾けた時の手前/奥の距離比 | 1.25 | 1.37 | 1.92 |
| マーカー検出限界距離 | ≈0.8m | ≈1.2m | ≈2.3m |

- 大きいボードの本質的な利点は「遠くから撮れる」ことではなく、**同じ距離で視野占有率が上がり、傾けたときの遠近情報が増える**こと
- 近距離手持ち運用（安定検出トリガー）では **A3 が実用的なベストバランス**
- A1 は全体を画面に収めるのに 0.75m 以上必要（はみ出し自体は ChArUco では問題なし）。45° 傾けると奥行き差 34cm でフォーカスが片側で甘くなる可能性あり。部屋規模の外部キャリブレーション用に有効

## 4. 撮影後の合格判定（fx が決まったかの検証）

RMS 再投影誤差は fx の信頼性指標にならない。以下で検証する:

### 4.1 fx 固定スイープ

fx/fy を自由推定値の ±4%・±8% に固定（`cv2.CALIB_USE_INTRINSIC_GUESS | cv2.CALIB_FIX_FOCAL_LENGTH`、主点・歪みは自由）して再キャリブレーションし、RMS の悪化を見る:

```python
flags = cv2.CALIB_USE_INTRINSIC_GUESS | cv2.CALIB_FIX_FOCAL_LENGTH
K0 = numpy.array([[f_fixed, 0, w/2], [0, f_fixed, h/2], [0, 0, 1.0]])
rms, K, d, _, _ = cv2.calibrateCamera(obj_list, img_list, (w, h), K0.copy(), None, flags=flags)
```

- **合格**: ±8% 固定で RMS が自由推定の 1.5〜2倍以上に悪化する（改善データ: 約2倍）
- **不合格**: ほぼ悪化しない（失敗データ: 9% 差で +11%）→ データは fx を決められていない

### 4.2 split-half 検証

キャプチャを前半/後半（および偶数/奇数）に分けてそれぞれキャリブレーションし、fx を比較する:

- **合格**: fx の差が 1% 以内（改善データ: 0.4% 以内）
- **不合格**: 数%振れる（失敗データ: 2.6%）

### 4.3 外部突き合わせ（可能なら)

過去のキャリブレーション結果や、別手法（基準点 + K 自由推定）の fx と比較する。スペック理論値（focal_mm / pixel_pitch_mm、本カメラでは 3.5/0.003 = 1166.7）から大きく（±10% 超）外れた場合も疑う。

## 5. ガイダンス機能化する場合の実装要素（実装示唆・未確定）

1. **リアルタイム傾き表示** — BoardDetector の検出結果（image_points / object_points）から solvePnP でボード法線を推定し、傾き角をライブビューに表示（「傾き: 38° ✓」）。カメラ設定変更は不要で、Calibration タブの設計ルール（カメラ設定変更禁止）に適合
2. **ポーズチェックリスト** — 「位置（中央/四隅/四辺）× 傾き方向（正対/左/右/上/下/対角）」のマトリクスを消し込み表示し、不足ポーズを誘導。安定検出トリガー（feat-009）と組み合わせて自動消し込み
3. **カバレッジヒートマップの拡張** — 現行の feat-010 は「位置」のみで本問題を検出できない。傾き分布（例: 「傾き30°+: 15/50枚」、ヒストグラム）を併記する
4. **キャリブレーション後の fx 感度自動診断** — 計算完了時に §4.1 / §4.2 を自動実行し、不合格なら「ポーズ多様性不足のため焦点距離が信頼できません。ボードを傾けたキャプチャを追加してください」と警告。`tools/offline_calibration.py` には `--fx-sensitivity` オプションとして搭載

実装する場合は独立性の高い 4（オフラインで完結・実機不要）→ 1 → 2/3 の順に分割して進めるのが、開発方針（小さく作って積み重ねる）に沿う。
