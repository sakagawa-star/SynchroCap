# SynchroCap Raw データ ことはじめガイド

SynchroCap で録画した `cam{serial}_{開始フレーム:06d}.raw` を、初めて扱う人向けに説明します。
「これは何のデータか」「中身はどうなっているか」「どう使うか」を順に解説します。


## 1. これは何のデータか

複数の産業用カメラを PTP 同期して録画したときに出力される、カメラの生（raw）映像データです。

普通の動画ファイル（MP4 など）と違い、カメラのセンサーが捉えた信号をほぼそのまま記録しています。

- 圧縮による画質劣化がない（研究・解析向き）
- 各フレームに撮影タイムスタンプ（ナノ秒）が埋め込まれている（カメラ間の同期解析向き）
- 普通の動画プレーヤーでは開けない（専用ツールが必要）


## 2. ファイル名の読み方

書式は `cam{シリアル}_{開始フレーム:06d}.raw` です。

例: `cam05520125_000000.raw`

- `05520125` … カメラのシリアル番号
- `000000` … このファイルの先頭フレーム番号（6 桁ゼロ埋め）

1 回の録画（セッション）で、カメラごとにファイルが作られます。


## 3. ファイルは分割されている

1 ファイルにつき 1000 フレームずつに分割保存されます。
ファイル名の数字は「そのファイルが何フレーム目から始まるか」を表します。

- `cam05520125_000000.raw` … フレーム 0 から 999
- `cam05520125_001000.raw` … フレーム 1000 から 1999
- `cam05520125_002000.raw` … フレーム 2000 から 2999

なお、最後のファイルは 1000 フレーム未満になることがあります
（総フレーム数が 1000 の倍数でないとき）。これは異常ではありません。

1 台のカメラの全映像を見るには、これらを順番につなげて扱います。
（後述のツールが自動でつなげてくれます）


## 4. 一緒に作られる CSV ファイル

録画すると、.raw ファイルとは別に、カメラごとに 1 つの CSV ファイルが作られます。

- ファイル名: `cam{シリアル}.csv`（例: `cam05520125.csv`）
- .raw と違い分割されず、1 カメラにつき 1 ファイルです

中身は 2 列のシンプルな表で、全フレーム分の行が並びます。

| 列名 | 内容 |
| --- | --- |
| frame_number | カメラのデバイスフレーム番号（4 桁ゼロ埋め。カメラ由来のため 0 始まりとは限らない） |
| device_timestamp_ns | 撮影タイムスタンプ（ナノ秒） |

### .raw との関係（注意）

CSV の timestamp（device_timestamp_ns）は、.raw ファイルの各フレームヘッダ（後述）の
timestamp_ns と同じ値です。CSV はこのタイムスタンプを表計算ソフトなどで手軽に開けるように
した補助ファイルです。

ただし frame_number 列には注意してください。これはカメラが付けるデバイスフレーム番号で、
.raw の frame_index（ホスト側で 0 から数える連番。後述）とは別物です。値も形式も異なるため、
この 2 つを同じ番号として突き合わせないでください。突き合わせに使えるのはタイムスタンプです。

### では必要ないのか

映像の再生・MP4 変換・解析だけなら、.raw ファイルだけで完結します
（タイムスタンプは .raw の中にあるため）。CSV が無くても困りません。

ただし CSV には次の利点があるので、削除せず一緒に保管しておくことを勧めます。

- 録画が壊れていないかの検証に使える。validate コマンドが .raw と CSV を突き合わせてチェックする。
  CSV が無い場合、その照合チェックは失敗ではなくスキップされる（安全網が 1 枚減る）。
- Excel などでタイムスタンプをすぐ確認・解析できる（バイナリを解析しなくてよい）。


## 5. ファイルの中身（全体構造）

ここからは .raw ファイル本体の中身の話です。独自の SRAW 形式で、構造はシンプルです。

1. ファイルヘッダ（40 バイト） … ファイルの先頭に 1 つだけ
2. フレーム 1 = フレームヘッダ（24 バイト）＋ 画像データ
3. フレーム 2 = フレームヘッダ（24 バイト）＋ 画像データ
4. 以降、フレームの繰り返し

つまり「先頭に説明書きが 1 つ、あとは（見出し＋画像）のセットが延々と続く」形です。


## 6. ファイルヘッダの中身（先頭 40 バイト）

ファイルに 1 つだけ置かれる、全体の説明情報です。

| 位置(byte) | サイズ | 型 | 項目 | 内容 |
| --- | --- | --- | --- | --- |
| 0 | 4 | 文字列 | magic | 固定値 "SRAW"（ファイル識別子） |
| 4 | 4 | uint32 | version | フォーマットバージョン（現在 1） |
| 8 | 16 | 文字列 | camera_serial | カメラシリアル番号（null 終端 ASCII） |
| 24 | 8 | int64 | recording_start_ns | このファイル先頭フレームの撮影時刻（ナノ秒）。下の注記参照 |
| 32 | 2 | uint16 | width | 画像の幅（ピクセル） |
| 34 | 2 | uint16 | height | 画像の高さ（ピクセル） |
| 36 | 2 | uint16 | pixel_format | ピクセル形式（下表参照） |
| 38 | 2 | uint16 | reserved | 予約領域（0） |

すべてリトルエンディアンです。

recording_start_ns は名前に「開始」とありますが、実際に入るのは「そのファイルの先頭フレームの
撮影時刻」です。録画は 1000 フレームごとにファイル分割されるため、先頭ファイル（frame 0 から始まる）は
ほぼ録画開始時刻になりますが、2 つ目以降のファイル（frame 1000 から、など）はそのファイル先頭フレームの
時刻であって録画開始時刻ではありません。ファイルをまたいだ時間の基準には、各フレームヘッダの
timestamp_ns を使ってください。


## 7. フレームヘッダの中身（各フレーム先頭 24 バイト）

画像 1 枚ごとに、その手前に置かれる見出し情報です。

| 位置(byte) | サイズ | 型 | 項目 | 内容 |
| --- | --- | --- | --- | --- |
| 0 | 4 | 文字列 | magic | 固定値 "FRAM"（フレーム識別子） |
| 4 | 4 | uint32 | payload_size | 続く画像データのバイト数 |
| 8 | 8 | uint64 | frame_index | フレーム番号（0 から始まる連番） |
| 16 | 8 | int64 | timestamp_ns | 撮影タイムスタンプ（ナノ秒） |

このフレームヘッダの直後（24 バイト目から）に、画像データ本体が `payload_size` バイト続きます。

ポイントは、各フレームに「何番目のフレームか」と「いつ撮影されたか（ナノ秒）」が
記録されていることです。これがカメラ間の同期解析を可能にしています。


## 8. 画像データの中身（重要な注意点）

画像データの形式は pixel_format で決まります。

| 値 | 形式名 | ビット深度 | バイト/ピクセル |
| --- | --- | --- | --- |
| 0 | BayerGR8 | 8bit | 1 |
| 1 | BayerGR16 | 16bit | 2 |
| 2 | BGR8 | 8bit | 3 |

通常の録画は BayerGR8（値 0）です。

1 フレームのバイト数は次式になります。

    payload_size = width × height × バイト/ピクセル

### Bayer とは

Bayer 配列はカラーセンサーの生データで、各画素は R/G/B のうち 1 色しか持っていません。
これをカラー画像に直すには「デモザイク（補間）」という変換が必要です。

### つまずきやすい点（赤と青が入れ替わる）

このデータを OpenCV でカラー変換するとき、名前は BayerGR8 なのに、
使う変換コードは COLOR_BayerGB2BGR（GR ではなく GB）です。

    bgr = cv2.cvtColor(bayer, cv2.COLOR_BayerGB2BGR)

これは GenICam（カメラ規格）と OpenCV で Bayer 配列の命名基準が違うためです。
間違えると赤と青が入れ替わった画像になります。本プロジェクトでも過去に修正された箇所です。


## 9. 実際の使い方（専用ツール）

中身を見たり変換したりするには、付属の `tools/raw_tool.py` を使います。

前提:

- コマンドはリポジトリのルートディレクトリで実行します。
- 録画ファイルは、録画スクリプトを実行したフォルダの下の `captures/<セッション日時>/` に
  作られます（例: `captures/20260608-101530/`）。このフォルダが「セッションフォルダ」です。
  下の例の `<セッションフォルダ>` や .raw ファイルのパスは、この実際のパスに読み替えてください。
- SynchroCap の Python 環境で実行してください（view は opencv-python、encode は ffmpeg が
  必要です。無い場合はエラーで終了します）。
- 引数の種類: dump と view には .raw ファイルのパスを、validate / sync-check / encode には
  セッションフォルダ（録画ファイル一式が入ったフォルダ）のパスを渡します。

ヘッダ情報を表示（解像度・フレーム数・タイムスタンプ確認）:

    python tools/raw_tool.py dump <セッションフォルダ>/cam05520125_000000.raw

セッションの整合性チェック（壊れ・欠落の確認。.raw と CSV を突き合わせる）:

    python tools/raw_tool.py validate <セッションフォルダ>

カメラ間の同期ズレを確認:

    python tools/raw_tool.py sync-check <セッションフォルダ>

1 フレームを画像で表示（デモザイク済み）。ウィンドウが開くため画面表示できる環境が必要です。
ウィンドウ上で q または ESC キーで閉じ、s キーで同じフォルダに PNG 保存できます:

    python tools/raw_tool.py view <セッションフォルダ>/cam05520125_000000.raw --frame 0

普通に見られる動画（MP4）に変換:

    python tools/raw_tool.py encode <セッションフォルダ> --serial 05520125 --fps 30

出力は同じセッションフォルダ内に cam{シリアル}.mp4 という名前で作られます。
同名ファイルが既にある場合はエラーになるため、先に退避または削除してください。
画質は --qp（0〜51、小さいほど高画質、既定 20）で調整できます。


## 10. 初めての人へのおすすめ手順

1. dump でファイルの素性を確認する（解像度・フレーム数）
2. view で 1 枚絵を見る（ちゃんと写っているか確認）
3. 全体を見たいなら encode で MP4 にする


## 11. 補足: 画質を落とさず見たい場合

注意: 本ツールの encode は HEVC（圧縮あり）の MP4 専用で、FFV1 への変換機能はありません。

研究の解析用途で画質を一切落とさず見たい場合は、圧縮ありの MP4 ではなく FFV1 という
ロスレス形式が適しています。これは raw_tool.py ではなく、ffmpeg を直接使って変換します
（生の bayer_grbg8 データを ffmpeg に渡し、-c:v ffv1 で .mkv に出力する）。
FFV1 の再生には VLC などのプレーヤーが必要です（OS 標準プレーヤーでは開けません）。
